GIT

GIT is a global information tracker.I highly recommend you to google it to get to know about various other funny and strange abbreviations git has! Now, coming to the pronunciation g in git is pronounced as "ghee"... except single e in that meaning pronounced as "ghe" and "it" is pronounced as just regular "it" nothing special about that

What is purpose of Git?

GIT is a tool that keeps track of various versions of your project and hence known as VCS (Version Control System). Also, it is open-source and free to use!!
Speaking of open-source and free to use: "It is developed by this great man Linus Torvalds" And yes he's the same person who gave us the "LINUX" which is also open-source and free to use Operating System.

Who uses Git?

  • Scientists: To collect various information and to study it
  • Writers: To keep track of different versions or parts in their stories
  • Government: Ammending laws, sharing other data etc
  • Engineers and Coders: They use it for keeping different versions or features in their projects (Well you might've thought only these category people use it :-) right? Not just the mentioned categories other people also use it for various purposes)

Git vs GitHub

Now this a common confusion in most of the people.They assume that both Git and GitHub are the same.

Example

Let's look at a simple scenario and try to understand the difference... Assume you are using Git to track various versions or features of your project. Now git runs in your local machine and only you can see all the history of changes you've made to your code and could keep track of it. Now if you want to share all those versions or changes to your friend or colleague who's located somewhere else what you would do? Ofcourse we can book a flight ticket and go to him/her with our laptop right ? What if I say there's a tool which saves your flight ticket!? Yes there is one and it is called "GITHUB".

GitHub

GitHub is a service which stores your local git repo (repo or repository is git workspace) on the cloud. And hence it requires internet to push your local folders into Github. It also helps in collaborating with other people. Now, inorder to collaborate other people should be able to identify that it's you right? Hence Github needs an account. Whereas git doesn't require internet or account. Just download it in your local machine and start working with it. And the limitation ?? It's already discussed: You won't be able to share your changes or collaborate with others if you're only using git and not github.

Now that we know what git is, let's see how we can use it in our projects to keep track of various versions.

Mostly git is used through command line interface (CLI) and it was first made and used it like that but as days passed by several GUI's (Graphical user interface) like GitHub Desktop, GitKraken etc were developed to interact and make it more user friendly. But still using git with CLI (Command line interface => terminal) has a separate fan base! And it's also good to start using git with CLI first as it gives good exposure in using CLI which is "THE MOST IMPORTANT SKILL."

You can check if git is installed on your machine using

git --version

That doesn't simply mean that you are using git. To use it and track various changes and history of all the actions that are done in that project we need to initialize a git repository in that project. We can have different repos and all these will not be connected.

Git commands

Let's see some of the git commands... inorder to start using git in your project go to the root directory of your project and execute

git init

This is the way to initialize a git repository which if done, git starts tracking your project. But it is WRONG! to execute this command first. It's always a good idea to first check if we had previously executed that command and check if a git repository is already initialized.

To check that, first check the status of git in your project using:

git status

You should get something like this:

"fatal: not a git repository (or any of the parent directories): .git"

That means git is not initialized in your project before and now you can go ahead and initialize it.

git init

This initializes empty git repository in your project which creates the .git "hidden folder" which has various subfolders which keeps track of all the history in our project.

But if you had initialized git more than once, do not panic its not the end of the world but git gets confused and git tracks about git so if we created nested repos fear not, and delete inner .git folder using:

rm -rf .git

Once you've initialized a git repository it starts tracking files and this tracking mechanism consists of two steps: Project files first must be added to the staging area and then commit.All the files that are in staging area will get committed.

Generally the flow is like this:
working directory->git add->staging area->git commit->repository
You can add all the untracked files into the staging area at once using:

git add .

To add specific files into the staging area.

git add filename1,filename2,...

To commit those changes we simply do:

git commit

Adds all the files in the staging area into git repo. but the above command pops a default editor which will be vim editor and asks for a message... instead we can directly give it as

git commit -m "first commit"

-m is the message flag and the message should be given inside double quotes
Now run git status which gives the list of files that are committed.

Some good practices for meaningful commit messages:

feat: to specify any info about any feature
fix: bug fix
docs: for changes in documentation
styling: everything related to styling
refactor: optimizing the repetitive code
chore: updating build tasks,package manager
testing: everything related to testing

Also it is a good practice to write command messages in present tense

Example

Let's say you added a new feature in your project and now you want to commit it. Your commit message should be something like:
feat: add new feature

git log command will list the log of all the files that are committed. It has commit hash, date and time at which commits are created author name and etc.

Also there are a lot other commands. But for a beginner it's good to atleast know these commands. I highly encourage you to google more and more about other git features and commands. What I just discussed is just a drop in the "git" ocean and I hope this really helped you understand basics of git and motivated you enough to explore more about it!
